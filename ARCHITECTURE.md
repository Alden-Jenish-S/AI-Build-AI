# AIBuildAI - Architecture & Workflow Reference

> **Project goal:** An autonomous multi-agent system that, given a tabular ML competition task, discovers, bootstraps, implements, evaluates, and reuses machine learning techniques through a memory-pool-augmented UCB1 tree search.

---

## 1. Directory Structure

```text
agent_system/
├── agents/
│   ├── manager_agent.py           # Orchestrates ablation/tree search and node execution
```

## 3. Runtime Configuration

### Task Config

Each task can include `task_config.json`:

```json
{
  "metric_name": "score",
  "metric_direction": "maximize",
  "subprocess_timeout": 300
}
```

`ManagerAgent` loads `metric_direction` and `subprocess_timeout` at startup. `ImplementationAgent` reads `metric_name` and requires generated code to write:

```json
{"score": 0.8123, "metric": "score", "direction": "maximize"}
```

Defaults are `metric_direction="maximize"` and `subprocess_timeout=300`.

### LLM Config

All agents call `agents/llm_utils.py`.

```bash
export NVIDIA_API_KEY="your-key"
# or
export GEMINI_API_KEY="your-key"

export LLM_MODEL="nvidia/nemotron-3-ultra-550b-a55b"  # optional
```

Provider behavior:

| Key present | Base URL | Provider default model |
|-------------|----------|------------------------|
| `NVIDIA_API_KEY` | `https://integrate.api.nvidia.com/v1` | `nvidia/nemotron-3-ultra-550b-a55b` |
| `GEMINI_API_KEY` | `https://generativelanguage.googleapis.com/v1beta/openai/` | `gemini-2.5-flash` |

`LLM_MODEL` or a constructor `model_name` overrides the provider default. Token usage is tracked globally for ablation metrics. Calls use a 120-second HTTP timeout and retry rate-limit failures up to 5 times with backoff starting at 10 seconds.

## 4. Memory Pool

The memory pool is a two-level library of reusable tabular ML techniques.

### L1 Index

`memory_pool/l1_index.json` maps categories such as:

| Category | Purpose |
|----------|---------|
| `gbdt_ensembling` | XGBoost, LightGBM, CatBoost fold ensembles |
| `feature_engineering_tabular` | Scaling, polynomial features, interaction features |
| `categorical_encoding` | Target and frequency encoding |
| `cv_strategies` | Stratified and task-aware validation splits |
| `blending_stacking` | Rank blending and stacking |
| `missing_value_imputation` | Median/mode or typed imputers |
| `factorization_machines` | Pairwise interaction models |
| `interaction_aware_gbdt` | Interaction construction plus boosting |
| Dynamic categories | Empty or newly discovered categories can be populated by L2Builder |

### L2 Artifacts

Each artifact in `memory_pool/l2_store/<category>/` has:

- `<artifact_id>.json`: model card with `artifact_id`, `category`, `description`, `interface.entrypoint`, `dependencies`, `verified`, `verification_log`, and `code_path`.
- `<artifact_id>.py`: self-contained implementation imported by generated node code.

### Query API

```python
from memory_pool.query_tool import query

query("gbdt_ensembling")                         # summaries only
query("gbdt_ensembling", "catboost_10fold_ensemble")  # full card + code_content
```

## 5. Task Workflow

Run an ablation:

```bash
cd agent_system
python eval/run_ablation.py tabular-playground-series-may-2022
```

### Phase 0 - Baseline Cache and Ablation Setup

`run_ablation.py` creates one deterministic baseline, caches its generated `initial_dataloader.py` and `initial_algorithm.py`, executes the baseline once to derive `baseline_score`, then restores the same starter files for every condition:

| Condition | Pool | Tree |
|-----------|:----:|:----:|
| No-pool, single-agent | No | No |
| No-pool, tree-search | No | Yes |
| Pool, single-agent | Yes | No |
| Pool, tree-search | Yes | Yes |

Single-agent mode uses a linear Technique -> Setup -> Implementation path. Tree mode uses the UCB1 search loop.

### Phase 1 - Task Initialization

```text
ManagerAgent.initialize_task()
  -> InitialAgent.generate_initial_code(task_dir)
       -> DataAnalyzer profiles train/test data
       -> LLM generates initial_dataloader.py
       -> LLM generates initial_algorithm.py
```

The generated baseline is based on `task_description.md` and dataset profiles, not hardcoded model logic.

### Phase 2 - Technique Selection

```text
TechniqueAgent.run(task_description, branch_plan, global_memory_context, l1_index)
  -> uses branch_plan to select at most 2 L1 categories
  -> queries L2 summaries for selected categories
  -> pool hit: returns full model card and code metadata
  -> pool miss: generates a clean methodology-focused web query
                searches web
                asks LLM for a raw technique outline
```

The clean task description is separated from the internal branch plan so search queries do not leak branch-bias wording or irrelevant file names.

### Phase 3 - Dynamic Bootstrapping

When a technique is a pool miss:

```text
L2Builder.build_from_source(..., commit=False, target_dir=node_dir)
  -> extracts model card and code from the raw outline
  -> writes local artifact files into node_dir
  -> SandboxVerifier runs the entrypoint in an isolated subprocess with mock data
  -> success: status becomes pool_miss_pending
  -> failure: node falls back without committing anything globally
```

A locally bootstrapped artifact is committed to the global memory pool only after the implementation run succeeds and beats the baseline according to the configured metric direction.

### Phase 4 - Implementation

```text
ImplementationAgent.run(node_dir, tech_record, task_dir)
  -> copies initial_dataloader.py into node_dir
  -> copies selected artifact code into node_dir when available
  -> links or copies task data into node_dir/input/
  -> reads or creates dataset_analysis_report.txt
  -> asks LLM to modify initial_algorithm.py to import and use the technique
  -> executes algorithm.py with the configured timeout
  -> on failure, runs a coder/debugging LLM loop up to 5 total attempts
  -> parses result.json or stdout for the validation score
```

Failed implementations return `score=None`, backpropagate zero reward, and do not spawn follow-up technique nodes.

### Phase 5 - Tree Search

Tree mode starts with a virtual root and three dynamic initial technique branches generated from the task description. The scheduler repeatedly selects the next unexecuted node by greedily walking the tree using UCB1:

```text
UCB1 = average_reward + c_t * sqrt(log(parent_visits) / node_visits)
```

Unvisited nodes score as infinity so each branch gets explored. The exploration constant decays by budget region:

```text
0%  -> 30% budget: c_t = 1.414
30% -> 70% budget: c_t decays toward 0.5
70% -> 100% budget: c_t = 0.5
```

Technique nodes receive a small reward (`0.01`) after resolving. Implementation nodes backpropagate `score` for maximize metrics and `-score` for minimize metrics. After each successful implementation, the manager asks the LLM for a short follow-up improvement plan and spawns a child technique node.

### Phase 6 - Best Node and Submission

After the budget is exhausted, `ManagerAgent` scans successful implementation nodes:

- `maximize`: choose the highest score.
- `minimize`: choose the lowest score.

`generate_final_submission()` copies the best node's `submission/submission.csv`, aligns it to `sample_submission.csv` when possible, and writes both:

- `tasks/<task_name>/submission.csv`
- `runs/<task_name>/submission.csv`

The manager also writes `runs/<task_name>/method_tree.png`.

## 6. Metrics

`eval/metrics.py` computes ablation metrics from the node history and derived baseline:

| Metric | Meaning |
|--------|---------|
| Pool Hit Rate | Technique nodes that reused a memory-pool artifact |
| Overcome Rate | Implementation nodes that beat the baseline |
| Medal Rate | Implementation nodes passing bronze/silver/gold thresholds |
| Gold Rate | Implementation nodes passing the gold threshold |
| Avg tokens/node | Total tracked LLM input/output tokens for the condition |

`eval/run_ablation.py` writes the final comparison table to `eval/results.md`.

## 7. Web Search Fallback Chain

```text
search_web(query)
  -> Serper API when SERPER_API_KEY is available
  -> DuckDuckGo Lite scraping with retries and rotating user agents
  -> DuckDuckGo Instant Answer API
  -> empty result; TechniqueAgent falls back to LLM internal knowledge
```

Search is used only when the selected memory-pool candidates are absent or not a strong match.

## 8. Design Invariants

| Invariant | Implementation |
|-----------|----------------|
| No hardcoded task algorithm | Baseline dataloader and algorithm are generated from task description plus dataset profile. |
| Fair ablation starter state | One generated baseline is cached and restored across all 4 ablation conditions. |
| Bounded retrieval | TechniqueAgent selects at most 2 L1 categories per branch. |
| Pool-first, novelty-aware selection | Strong pool matches are reused; weak/empty matches trigger web/LLM bootstrapping. |
| Verified bootstrapping | New technique code must pass local sandbox verification before execution. |
| Clean global memory pool | Dynamic artifacts are committed globally only after a successful, baseline-beating run. |
| Robust node execution | Node-level failures are isolated; failed implementation nodes receive zero reward and stop branching. |
| Debuggable implementations | ImplementationAgent can regenerate/fix code after subprocess failures. |
| Metric-aware search | Minimize metrics are converted to negative reward for scheduling while best-node selection honors the original direction. |
| Central LLM config | Provider selection, retries, model override, and token tracking live in `llm_utils.py`. |
| Submission compatibility | Best-node submissions are aligned to the sample submission schema when available. |
